1. Project Overview
Group 05 is tasked with building a production-grade Banking AI Copilot for the retail banking and BFSI sector. The
copilot assists customers and relationship managers with loan eligibility queries, KYC process guidance, fraud alert
explanations, account policy Q&A, and regulatory compliance information — all grounded in actual banking
documents and governed by strict regulatory guardrails.
The key technical differentiator for this project is Self-Reflective RAG — also known as Corrective RAG (CRAG).
Unlike standard RAG, the agent does not blindly use whatever chunks are retrieved. Instead, it grades the relevance
of each retrieved chunk, and if the retrieval quality is insufficient, it rewrites the query and re-retrieves — up to two
reflection iterations — before falling back to web search. This pattern is critical in banking where a low-quality retrieval
leading to a hallucinated policy answer can cause regulatory and reputational harm.

2. Problem Statement
Retail banks handle millions of customer queries daily — spanning loan eligibility, KYC documentation requirements,
interest rate queries, fraud dispute guidance, account terms, and regulatory disclosure obligations. The challenge is
not just answering these queries — it is answering them accurately and compliantly, with zero tolerance for
hallucinated policy terms, fabricated regulatory citations, or unsolicited financial advice that could expose the bank to
liability.
Standard RAG systems fail in banking when the retrieved context is marginally relevant but not precise enough — the
LLM fills the gap with plausible but wrong information. A customer told the wrong KYC document requirement wastes
time and damages trust. A customer given an inaccurate loan eligibility criterion may make a financial decision based
on false information. The Self-Reflective RAG pattern forces the agent to interrogate its own retrieval quality before
generating any response.
■ Your Mission
Build a banking AI copilot that uses a Self-Reflective RAG pipeline to ensure every response is grounded in
high-quality retrieved context. The agent must grade each retrieved chunk, rewrite the query if relevance is
insufficient, and only generate a response when it has verified the retrieval quality — or explicitly flag the response
as 'low confidence' and surface the web search fallback.
Every response must pass a regulatory compliance check before reaching the user. The system must demonstrate
observable self-reflection iterations in LangSmith traces — showing the grading step, the rewrite, and the
re-retrieval.

2.1 Key Pain Points Your System Must Address
  Loan Eligibility Confusion: Customers misunderstand eligibility criteria (credit score, income ratio, employment
 type) leading to wasted applications and agent time.
  KYC Document Uncertainty: Customers and agents frequently need clarification on which documents are
 required for different account types, customer profiles, or transaction thresholds.
  Fraud Alert Opacity: When an account is flagged or a transaction is blocked for fraud, customers receive minimal
 explanation — the copilot must explain the alert type and resolution steps.
  Interest Rate & Fee Queries: Customers cannot easily get accurate, cited answers on current interest rates,
 processing fees, penalty charges, and prepayment terms.
  Regulatory Disclosure Requests: Customers and compliance officers need access to RBI guidelines, Fair
 Practices Code, DPDPA data handling terms, and AML policy — buried across multiple documents.
  Hallucination Risk: In banking, a standard RAG system that returns a slightly off-topic chunk can generate a
 confident but wrong answer — e.g., citing a home loan LTV ratio for a personal loan query.
  Low-Confidence Query Handling: Queries on very recent regulatory changes or product updates may not be in
 the document corpus — the system must recognise this and flag it rather than guess.
■■ Non-Negotiable Regulatory Constraint
The copilot must NEVER provide personalised financial advice, approve or reject a loan application, or guarantee
interest rates. All regulatory citations must reference the actual document and clause number. The system must
include a mandatory disclaimer on every response: 'This information is for guidance only. Please consult your
relationship manager or refer to official RBI/bank circulars for binding decisions.' Failure to implement this
disclaimer is an automatic deduction.

■ CRAG Design Hint
Your chunk grader must score each retrieved chunk on two dimensions: (1) Topical relevance — does this chunk
address the query? (2) Specificity — is this chunk precise enough to ground a safe answer, or is it generic policy
language that could mislead? Think carefully about your grading prompt and scoring threshold. The two CSVs
(interest rates, loan eligibility matrix) should be handled as structured lookups — not embedded in the vector store
— to prevent approximate matching returning wrong rate figures.

4.1 Pipeline Stage Responsibilities
 Stage
 Type
 Responsibility
1. Retrieve
RAG Retrieval
Initial FAISS semantic search returning top-k chunks. Metadata filters applied by
query intent (loan / account / KYC / fraud / regulatory). Web search tool available
as fallback source.
2. Grade
Documents
Self-Reflection
LLM grades each retrieved chunk on relevance (0–1 score) and specificity. Chunks
below the relevance threshold are discarded. Average score compared against a
configurable threshold to decide whether to generate or rewrite.
3a. Generate
LLM
Generation
Triggered when retrieved chunks pass the relevance threshold. LLM generates a
grounded answer citing specific chunks, clause references, and document names.
Answer explicitly labelled with confidence level.
3b. Rewrite Query
Query
Rewriting
Triggered when retrieved chunks fail the relevance threshold. LLM reformulates the
query with different terminology, broader scope, or more specific intent. Loop
repeats back to Step 1 (max 2 iterations). After 2 failed iterations, web search
fallback triggered.
4. Compliance
Check
Regulatory
Filter
Dedicated LLM call checks the generated answer against banking regulatory
constraints: no personalised financial advice, no guaranteed rates, no unsolicited
investment recommendations, mandatory disclaimer present, no fabricated
regulatory citations.
5. Guardrail Layer
Safety Filter
Screens for PII/PHI in output, prompt injection in input, credit score fabrication, and
loan approval language. Adds the mandatory regulatory disclaimer if missing.
